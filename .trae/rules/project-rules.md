# ezBookkeeping 项目开发规则

## 1. 项目架构规范

### 1.1 目录结构

#### 后端
- **cmd/**: 命令行工具和服务器入口
- **pkg/**: 核心功能包
  - **api/**: API接口
  - **auth/**: 认证功能
  - **core/**: 核心业务逻辑
  - **models/**: 数据模型
  - **services/**: 业务服务
  - **datastore/**: 数据存储
  - **utils/**: 工具函数
- **conf/**: 配置文件
- **docker/**: Docker相关文件

#### 前端
- **src/**: 源码
  - **components/**: 组件
    - **base/**: 基础组件
    - **common/**: 通用组件
    - **desktop/**: 桌面端组件
    - **mobile/**: 移动端组件
  - **core/**: 核心逻辑
  - **router/**: 路由配置
  - **styles/**: 样式文件
  - **locales/**: 国际化文件
- **public/**: 静态资源

### 1.2 模块划分原则
- **高内聚、低耦合**
- **关注点分离**
- **可测试性**
- **可扩展性**

## 2. 编码标准

### 2.1 命名约定

#### 后端（Go）
- **包名**: 小写字母
- **函数名**: 驼峰命名法，首字母大写表示公开
- **变量名**: 驼峰命名法，首字母小写表示私有
- **常量名**: 全大写，下划线分隔
- **结构体名**: 驼峰命名法，首字母大写

#### 前端（TypeScript/Vue）
- **组件名**: PascalCase
- **变量名**: camelCase
- **常量名**: UPPER_SNAKE_CASE
- **函数名**: camelCase
- **类型名**: PascalCase

### 2.2 代码风格

#### 后端（Go）
- 遵循 Go 语言标准风格
- 使用 `go fmt` 格式化
- 每行不超过 120 字符
- 函数不超过 50 行，超过则拆分

#### 前端（TypeScript/Vue）
- 使用 ESLint 检查
- 遵循 Vue 3 最佳实践
- 使用 Composition API
- 组件逻辑清晰，职责单一

### 2.3 注释要求
- **公共 API**: 必须添加文档注释
- **复杂逻辑**: 添加注释说明思路
- **魔法值**: 注释说明来源
- **临时方案**: 注释说明原因

## 3. 开发流程

### 3.1 分支管理
- **main**: 发布稳定版本
- **develop**: 集成新功能
- **feature/**: 开发新功能
- **bugfix/**: 修复 bug
- **release/**: 准备发布

### 3.2 提交规范
- 遵循 Conventional Commits 规范
- 使用提交信息生成器生成说明
- 提交前运行代码检查和测试

### 3.3 代码审查
- 所有变更通过 Pull Request
- 至少 1 名 reviewer 批准
- 审查重点：质量、安全性、性能、可维护性
- 确保测试通过

## 4. 技术栈规范

### 4.1 后端
- **语言**: Go 1.20+
- **数据库**: SQLite, MySQL, PostgreSQL
- **Web 框架**: 自定义 HTTP 服务器
- **认证**: JWT, OAuth2
- **存储**: 本地文件系统, MinIO, WebDAV

### 4.2 前端
- **框架**: Vue 3
- **语言**: TypeScript
- **UI 库**: Vuetify
- **状态管理**: Pinia
- **路由**: Vue Router
- **图表**: ECharts
- **日期选择**: Vue Datepicker

### 4.3 应用场景
- **存储**: 小文件使用本地文件系统，大文件考虑 MinIO
- **认证**: 内部用户使用 JWT，外部集成使用 OAuth2
- **地图**: 根据地区选择合适的提供商
- **AI 功能**: 仅在需要时启用，注意 API 密钥管理

## 5. 接口设计

### 5.1 API 命名
- RESTful 风格
- 资源名称使用复数
- 操作使用 HTTP 方法
- 路径参数使用 `{id}` 格式

### 5.2 请求/响应格式
- **请求**: JSON 格式
- **响应**: 统一格式 `{"success": true, "data": {...}, "error": null}`
- 错误响应包含错误码和信息

## 6. 错误处理

### 6.1 后端
- 使用自定义错误类型
- 统一错误响应格式
- 记录详细错误日志
- 对外部返回友好错误信息

### 6.2 前端
- 统一处理 API 错误
- 显示用户友好的错误提示
- 记录错误日志
- 实现错误边界

## 9. 项目约束

### 9.1 性能要求
- 轻量级设计，支持资源受限环境
- 响应时间合理

### 9.2 安全性要求
- 敏感信息加密存储
- API 访问控制

- 定期更新依赖库

### 9.3 国际化要求
- 支持多语言
- 文本资源集中管理
- 日期、时间、货币格式本地化

### 9.4 部署要求
- 支持 Docker 部署
- 配置文件外部化
- 支持多环境部署

## 11. 版本控制

### 11.1 版本号规范
- 语义化版本号：MAJOR.MINOR.PATCH
- MAJOR: 不兼容的 API 变更
- MINOR: 向后兼容的新功能
- PATCH: 向后兼容的 bug 修复

### 11.2 发布流程
- 从 develop 分支创建 release 分支
- 测试和 bug 修复
- 合并到 main 分支并打标签
- 发布版本

## 13. 理财管理模块规范

### 13.1 开发范围
- **界面开发**：暂时只考虑桌面版
- **多语言支持**：暂时只考虑中英文
- **界面风格**：保持与现有系统颜色一致
- **数据分析**：参考市面上的财富管理软件

### 13.2 功能要求
- **投资组合管理**：支持多种投资类型的管理
- **收益分析**：提供投资收益的详细分析
- **风险评估**：评估投资风险水平
- **报表生成**：生成投资相关的报表

### 13.3 技术实现
- **前端**：使用 Vue 3 + TypeScript + Vuetify
- **图表**：使用 ECharts 实现数据可视化
- **状态管理**：使用 Pinia 管理状态
- **路由**：集成到现有路由系统

### 13.4 界面设计
- **布局**：采用与现有系统一致的布局风格
- **颜色**：使用系统统一的颜色方案
- **组件**：复用现有组件，保持风格一致
- **响应式**：确保在不同屏幕尺寸下的良好显示

### 13.5 数据模型
- **投资产品**：股票、基金、债券等
- **交易记录**：买入、卖出、分红等
- **收益数据**：历史收益、当前收益等
- **风险指标**：波动率、夏普比率等
